Goal: pages contain ONLY components. Components contain ONLY sub-components.
This step is read and plan only. Code extraction happens in the `ui-extract-tdd` slug.

THE RULE: No direct JSX in page files.
  After extraction, a page/route file should import and compose components ‚Äî nothing else.
  Every distinct visual section in the page is its own component.
  Every component that renders multiple distinct sections should be split into sub-components.
  If a component has N visually distinct parts, it should compose N sub-components.

Checklist:
[ ] List reusable functions/components: `codeyam-editor editor glossary-list`

Read the glossary through the CLI, never with the `Read` tool.
  `.codeyam/glossary.json` exceeds the Read tool's token limit. Use
  `codeyam-editor editor glossary-find <name>` (flags: `--prefix`,
  `--substring`, `--feature`, `--format`) for a single lookup. Use
  `codeyam-editor editor glossary-list`,
  `codeyam-editor editor glossary-untested`, or
  `codeyam-editor editor glossary-by-tag <tag>` for projections across the
  whole table. If you need to grep the table directly, read
  `.codeyam/glossary-index.txt` (a regenerated, line-oriented sidecar ‚Äî see
  the `glossary_index_file` glossary entry).
[ ] Read EVERY file used by this page/feature
    For EACH file, identify EVERY extractable piece:
    Components: headers, nav, loading states, empty states, error states,
      badges/pills, image containers, card sections, form fields, modals,
      titles, descriptions, rating displays, status indicators, buttons
    Functions: data transforms, calculations, formatting, validation,
      API response shaping, any logic that is not directly about rendering
    Hooks: data fetching, state management, side effects

[ ] Write a numbered extraction plan listing EVERYTHING you will extract
    The end state: every page file is ONLY imports + component composition.
    No raw <div>, <span>, <h1>, <p>, <img>, or <ul> in page files.
    Every visual section in every component is itself a component.

    For each item in the plan, note:
    ‚Äî What it is (component, function, hook)
    ‚Äî Where it currently lives (source file + approximate lines)
    ‚Äî Where it will go (new file path)

Present the numbered plan, then proceed to the `ui-extract-tdd` slug to execute it.

Pre-Deletion Sweep:
  Before deleting files or removing public functions, build a complete
  reference list in a single upfront sweep ‚Äî don't discover callsites
  incrementally through compile errors and test failures.

  1. List every symbol, file path, and module name being removed.
  2. Run ONE parallel batch of `Grep` calls (output_mode: files_with_matches)
     for each name across the whole workspace ‚Äî source, tests, docs, config
     JSON, `.gitattributes`, merge-driver setup, CI files.
  3. Read the hits together. Decide what must be removed, updated, or left
     alone.
  4. THEN make the deletions, updating every call site in the same pass.

  This replaces the delete ‚Üí compile-fail ‚Üí grep ‚Üí test-fail ‚Üí grep-again
  cycle with a single upfront pass. Especially important for:
    - Public functions used across crates or packages
    - Merge driver entries (`.gitattributes`, `.git/config`, scripts in
      `.codeyam/`)
    - Test helpers and fixtures that look unused but are called from other
      test files
    - Glossary entries (see next section)

  If you catch yourself running a fifth targeted grep after a deletion
  failed, stop and do the sweep properly.

Glossary Staleness:
  Before running `codeyam-editor editor audit --summary-only` at the audit gate, clean up
  glossary entries for code you've deleted ‚Äî the audit gate blocks
  advancement on stale entries.

  In the same pass as the pre-deletion sweep, for each removed function or
  component:
    1. codeyam-editor editor glossary-find <name>   ‚Äî confirm the entry exists
    2. codeyam-editor editor glossary-remove <name> ‚Äî remove it

  Treat stale glossary entries as part of the deletion, not a follow-up
  cleanup triggered by a blocked audit.
